# Automated Verification

[English](README.md) | [简体中文](README.zh-CN.md)

Use this when AI changed the code and you need more than "it seems fine."

## What this solves

AI can change code faster than people can manually re-check product behavior. Without a repeatable check, every follow-up edit can quietly break the same path again.

This scenario helps you create just enough evidence before review, merge, or release.

## Use it when

- AI changed UI, API, mobile, or workflow logic.
- Manual checking is slowing review down.
- A core user path could break.
- You need evidence before merge or release.

## Mini tutorial

1. Pick one behavior to protect.
   Choose the path users would notice first if it broke.

2. Decide the cheapest layer to test.
   If the risk is UI state, use a browser or mobile flow. If the risk is business logic, an API or unit-level check may be faster.

3. Make the check fail on purpose once.
   A test that cannot fail is decoration.

4. Run it before AI work.
   Establish the baseline before asking an assistant to change related code.

5. Save the result.
   A trace, screenshot, log, or CI run makes review easier than "I tried it locally".

## Example smoke check

```md
Behavior:
Workspace admin can invite one teammate.

Check:
1. Open workspace settings.
2. Submit a valid email.
3. Confirm the invite appears as pending.
4. Confirm duplicate invite shows an error.

Evidence:
- Playwright trace or manual screen recording.
- Test command output.
```

## Common paths

- Web UI checks.
- Mobile E2E checks.
- API regression checks.
- AI output evaluations.
- CI checks before merge.

## Tool fit

Use Playwright when browser behavior matters. Use Maestro for mobile flows. Use API checks when the UI is not the risky part. Use eval tools when output quality is subjective or generated by an LLM.
