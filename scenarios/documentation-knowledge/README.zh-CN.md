# 文档与知识

[English](README.md) | 简体中文

当决策和项目知识总是消失在聊天里，就看这个场景。

## 解决什么问题

团队经常在 PR、事故线程和聊天里做出不错的决策。三个月后，没人记得当时为什么这么选。

这个场景帮你把重要决策和经验变成轻量知识，让以后的人还能找到。

## 什么时候用

- 架构或产品决策之后还会被用到。
- 新队友缺少项目背景。
- 文档过期、分散，或者更新成本太高。
- 最近一次工作里的经验应该变成可复用知识。

## 小教程

1. 在离工作最近的地方记录决策。
   趁上下文还新鲜时写。

2. 写取舍，不只写结论。
   未来读者需要知道为什么选这条路，而不是另一条路。

3. 放到人真的会找的地方。
   工程决策适合 repo docs。跨团队背景更适合团队知识库。

4. 保持小。
   一次决策，一页说明。太长的文档更容易被忽略。

5. 加维护提示。
   告诉未来读者什么时候应该重新看这个决定。

## 示例决策笔记

```md
决策:
Workspace invite actions 使用 server-side permission checks。

原因:
Client-side checks 可以改善 UX，但不能作为授权依据。

取舍:
UI 仍然会隐藏不可用操作，但 server 才是权限判断的准绳。

什么时候重看:
Workspace permission model 发生变化。
```

## 常见路径

- 决策记录。
- 架构笔记。
- Runbooks。
- Onboarding docs。
- 个人知识系统。

## 工具怎么选

和 repo 绑定的工程决策适合 docs-as-code。运维说明和跨团队背景适合团队 wiki。还没准备好成为团队 guidance 的研究，先放个人笔记。
