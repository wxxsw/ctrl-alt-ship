# Requirements to Tasks

[English](README.md) | [简体中文](README.zh-CN.md)

Use this when a request is real, but still too vague for an engineer or AI assistant to start safely.

## Use it when

- A request can be interpreted in several ways.
- You need agreement on what "done" means.
- The work may touch product behavior, design, data, or rollout.
- You want a small, reviewable task before code starts.

## Common paths

- Turn a loose idea into a short spec.
- Turn a PRD or issue into an implementation plan.
- Turn a small request into an AI-ready task.
- Check a design flow before writing code.
- Plan a brownfield change before touching existing systems.

## First move

Write a short task brief:

- What should change?
- What should not change?
- How will someone know it is done?
- What needs to be checked before merge?
