# 需求到任务

[English](README.md) | 简体中文

当一个需求已经真实，但还模糊到工程师或 AI assistant 不能安全开工，就看这个场景。

## 解决什么问题

很多 AI 写代码失败，不是失败在代码里，而是失败在开工前。口头听起来很清楚的需求，一到实现就开始漂，因为没人写清行为、边界和验收标准。

这个场景帮你把松散意图变成一小块可以 review 的工作。

## 什么时候用

- 一个请求可以被理解成好几种做法。
- 几个人需要先对"完成"达成一致。
- 工作可能牵涉产品行为、设计、数据、权限或发布。
- 你希望 AI assistant 不靠猜来补上下文。

## 小教程

1. 先用一句话写清用户能看到的变化。
   例如："团队管理员可以在工作区设置里通过邮箱邀请成员。"

2. 写清边界。
   哪些包括，哪些不包括。很多需求失控，就是因为这里没写。

3. 写验收检查。
   用能观察到的语言，比如："邀请邮件已发送"、"成员列表出现待处理邀请"、"重复邀请会提示错误"。

4. 给实现线索。
   点出可能相关的页面、API、数据模型或命令。不要替工程师写代码，但要给第一轮调查足够方向。

5. 写验证方式。
   小功能可以先手动检查。如果这个路径以后经常改，最好补一条冒烟测试。

## 示例任务卡

```md
意图:
工作区管理员可以通过邮箱邀请一位成员。

不做:
- 不做批量邀请。
- 不改角色管理。
- 不碰计费逻辑。

验收:
- 管理员可以提交一个合法邮箱。
- 成员列表里出现待处理邀请。
- 重复邀请会显示清楚的错误。
- 非管理员用户不能访问邀请操作。

验证:
- 跑 workspace members 相关的现有单测。
- 在浏览器里手动做一次邀请流程冒烟测试。
```

## 常见路径

- 松散想法变成短 spec。
- PRD 或 issue 变成实现计划。
- 小需求变成 AI-ready task。
- 设计流程变成工程检查清单。
- Brownfield 改动变成影响说明。

## 工具怎么选

很多时候，普通 Markdown 已经够用。只有当工作跨团队、需要反复 review，或者必须和更大的 roadmap 连接时，才需要更重的 spec 或 planning 工具。
