# Editorial Guidelines

These guidelines keep Ctrl Alt Ship from becoming a generic AI tools directory.

## Write For A Moment Of Work

Assume the reader is in the middle of a real engineering task. They do not want a long essay first. They want orientation, decision help, and a next step.

Good opening:

> AI can generate code quickly. This scenario helps you prove whether the generated change actually works.

Weak opening:

> Testing has been an important part of software engineering for decades...

## Prefer Decisions Over Descriptions

Every page should help the reader choose.

Use:

- Recommended when...
- Avoid when...
- Start with...
- Upgrade to...
- Check this before trusting it...

Avoid:

- Comprehensive history
- Tool marketing copy
- Long feature lists
- Generic "pros and cons" without scenario fit

## Name The Negative Space

Every recommendation should include when not to use it.

This is a core project norm. It builds trust and prevents hype.

## Keep Weekly Updates In Their Lane

Weekly radar updates are useful because tools change quickly. But the radar should not become the product.

Radar items should:

- Point back to affected scenarios.
- Explain practical impact.
- Recommend an action.
- Link to sources.

Radar items should not:

- Become general AI news.
- Mention launches with no engineering workflow impact.
- Replace updates to scenario pages when a recommendation truly changes.

## Tool Inclusion Criteria

A tool can be added when at least one is true:

- It is a strong default for a scenario branch.
- It solves a recurring engineering bottleneck.
- It changes how a workflow should be done.
- It is a useful alternative with clear tradeoffs.
- It is widely used enough that readers need a decision note.

A tool should not be added only because it is new, popular, funded, or impressive in a demo.

## Evidence Standard

Prefer evidence from:

- Official documentation
- Release notes
- Reproducible examples
- Real project usage
- Public case studies
- Maintainer statements

If evidence is weak, say so.

## Tone

Practical, skeptical, and helpful.

The project can be lightly playful, but the content should make engineers feel oriented rather than entertained at the expense of clarity.
