# Project Charter

Ctrl Alt Ship exists to help software engineering roles use AI through real work scenarios, not through tool hype.

## Origin

The project began from a simple observation:

There are many AI tools for developers, but developers do not usually start with "which tool should I browse today?" They start with a real problem:

- I need to clarify a fuzzy requirement.
- I need an agent to understand my project.
- I need to verify AI-generated changes.
- I need to debug a production issue.
- I need my team to use AI safely and consistently.

Ctrl Alt Ship turns those recurring engineering situations into stable entry points.

## One-Sentence Vision

面向真实研发场景的 AI 工作方法地图：找到场景，选对工具，有凭有据地 ship。

## English Positioning

A scenario-first map for AI-powered engineering workflows: find the scene, pick the stack, ship with receipts.

## What We Are Building

We are building a durable, open-source map of AI-native engineering work.

The map has two layers:

- Stable layer: engineering scenarios that do not change every week.
- Moving layer: tools, workflow patterns, examples, and weekly radar updates.

The project should help a reader move from "I am stuck in this engineering situation" to "Here is the workflow I should try, the tools worth comparing, and the evidence I should require."

## What We Are Not Building

Ctrl Alt Ship is not:

- An awesome list of AI tools.
- A blog archive.
- A vendor directory.
- A ranking site.
- A prompt collection.
- A weekly AI news feed.
- A replacement for official documentation.

Tools are included only when they help a scenario. Radar updates exist only to keep scenario recommendations fresh.

## Target Readers

Primary readers:

- Individual developers who want practical AI workflows.
- Tech leads who need repeatable engineering practices.
- QA and test engineers who need better verification loops.
- Engineering managers who need team-level AI norms.
- Platform engineers who manage agent permissions, CI, MCP, and tooling.

Secondary readers:

- AI product engineers building LLM features.
- Developer advocates explaining practical adoption paths.
- Open-source maintainers evaluating AI-assisted contribution workflows.

## User Promise

When a reader opens this repo, they should be able to:

1. Recognize a familiar engineering scenario quickly.
2. Understand why the scenario matters.
3. See branches inside the scenario, such as web, mobile, API, or AI evals.
4. Pick a minimal workflow to try.
5. Compare relevant tools without drowning in options.
6. Know when not to use the recommended path.
7. Find recent updates without losing the stable map.

## Success Criteria

The project is working if:

- A new reader can find the right scenario in under 30 seconds.
- A scenario page gives a useful first action in under 5 minutes.
- Contributors add scenario branches rather than random tool entries.
- Weekly updates strengthen scenario pages instead of creating a news swamp.
- Readers learn how to choose and verify workflows, not just collect links.

## Editorial North Star

Every page should answer a version of this question:

> In this engineering situation, what work method should a capable developer try next, and how can they know whether it helped?
