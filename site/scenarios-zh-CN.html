给人看的场景教程

先看场景，再选工具。

这不是工具榜，也不是给 AI 的任务规划。它更像一张工程师的路线图：你现在卡在哪里，第一步该做什么，哪条路线适合，什么时候该停。

需求到任务 项目上下文记忆 编码与重构 调试 自动化验证 代码审查与质量门禁 文档与知识 发布与变更管理 事故响应 团队 AI 治理
01

场景教程

需求到任务

需求是真的，但还不够稳。

很多工作都是这样开始的：“能不能顺手加一下？”大家觉得听懂了，结果一开工，角色、空状态、计费、灰度、文案全冒出来。这个场景不是让你写大文档，而是先把活儿收成一块能做、能验、能 review 的任务。

先做这一步: 先写一句用户能看到的变化，再写清这次不做什么。
写清变化圈住边界列验收检查指向相关代码定验证方式

轻量 issue 任务卡

小修、小功能、预期行为清楚时用。直接写在 issue 或 PR 里。

PRD 拆任务

方向已经定了，但工程侧还需要切片和排顺序时用。

验收例子

角色、状态、边界条件会影响结果时用。

AI 辅助梳理

让 AI 帮你找缺口可以，别让它替你拍板产品决策。

管理员可以通过邮箱邀请一位成员。不做批量邀请，不改角色，不碰计费。成功标准是成员列表出现待处理邀请，重复邀请会给出清楚错误。
别踩这个坑: 最容易出事的地方，是任务看起来写得很完整，但真正的决策还藏着。权限、定价、发布策略没定，就先别让任何人开写。

场景教程

项目上下文记忆

AI 能读文件，但它不一定懂这个项目。

如果每个新会话都要从“项目怎么跑、目录怎么看、哪里不能乱改”讲起，那问题不只是 AI 不聪明。这个 repo 需要一小块稳定记忆。

先做这一步: 写下最近五件你重复解释过的事，只留下稳定的部分。
收集重复解释筛稳定事实放在 repo 附近链接深层文档重复犯错就更新

Repo 指南

放启动命令、测试命令、目录结构、代码约定和 review 预期。

代码库地图

适合大 repo 或老系统，先告诉人该从哪里看。

决策记忆

记录以后还会被追问的架构和产品取舍。

Assistant 上下文

放团队认可的规则，不放个人 prompt、凭证和本地秘密。

pnpm install 安装，pnpm test 测试。app/ 放产品页面，api/ 放 server routes，db/ 放 schema 和 migrations。Billing 和 membership 需要额外 review。
别踩这个坑: 别把它写成私人笔记大合集。共享规则进 repo，个人 agent notes 和 credentials 留在 git ignore 里。

编码与重构

AI 写得很快，你要负责让改动还能被人 review。

实现、清理、迁移、重构都在这里。真正的问题不是“AI 能不能写代码”，它通常能。问题是写出来的 diff 还有没有形状，能不能被人看懂、验证、接住。

先做这一步: 让 AI 先给计划。计划里必须有相关文件、风险区域和验证方式。
从任务开始先确认计划按检查点改跑便宜检查对照任务 review

结对式 assistant

适合局部编辑、代码搜索、小 bug 修复和测试草稿。

Agentic loop

适合多文件编辑，并且能用命令行验证的任务。

Codemod 迁移

适合重复模式，能机械化就不要手工到处改。

先保护旧行为

适合老代码区域，行为不能悄悄变。

把 notification preference 逻辑抽到 hook。UI 行为、API shape、持久化都不能变。如果需要改 API，先停下来确认。
别踩这个坑: 最容易失控的是“小重构”变成漫游式 cleanup。diff 不再服务原任务，就停下来拆。

场景教程

调试

别让一个听起来合理的故事，直接变成修复。

你现在有一个现象：一条 stack trace、一张图的 spike、一个用户反馈。AI 在这里很有用，但前提是它帮你生成可验证的假设，而不是把猜测包装得更像答案。

先做这一步: 写下现象、证据，以及一个能预测下一个观察结果的假设。
写清现象收集证据分开猜测测试假设补回归检查

先复现

能本地复现或失败稳定时用。

先观察

只在生产出现，或系统链路很长时用。

查变更历史

发布、flag、配置或依赖变化之后出问题时用。

AI 生成假设

让 AI 给竞争假设，不要接受第一个说法。

使用 coupon 后 checkout 返回 500。日志显示 currency_code 是 null。假设：非默认币种下 coupon recalculation 丢了 currency。下一步用 EUR 和 USD fixture trace。
别踩这个坑: 一次改三处很像在推进，其实是在毁证据链。

场景教程

自动化验证

把“看起来没问题”变成证据。

AI 让改代码变便宜了。这是好事，直到改动跑得比证据还快。验证不是为了自动化一切，而是保护那些静悄悄坏掉会很疼的行为。

先做这一步: 先说清你要保护哪一个行为，再选最便宜且可靠的检查。
命名行为选择层级让它失败一次本地或 CI 跑附上证据

单测或组件测试

适合规则、解析、权限和 UI 状态。

浏览器或移动端 E2E

适合跨层的关键用户流程。

API 或 contract test

适合 API 就是产品边界的地方。

AI eval

适合 RAG、agent、抽取和输出有波动的功能。

管理员邀请成员：登录、进设置、提交邮箱、确认待处理邀请出现、重复提交、确认错误出现。失败时保留 trace。
别踩这个坑: E2E 很像真实世界，所以团队容易滥用。小测试能抓同样风险，就先用小测试。

场景教程

代码审查与质量门禁

先 review 意图，再欣赏 diff。

AI 可以生成很大、很顺眼的 diff。这不是让 review 更简单，而是更难。好的 review 先看任务，再看风险和证据，最后才看风格。

先做这一步: 先读任务卡。第一问不是代码漂不漂亮，而是 diff 还像不像原任务。
先读任务检查范围扫描风险看证据review 可维护性

人工 review checklist

看行为、范围、架构和判断。

静态检查

抓类型、lint、安全、依赖和重复问题。

Build 和 test gates

放那些必须挡住 merge 的检查。

AI 辅助 review

适合总结和找风险，不承担最终 ownership。

问自己：diff 是否实现已接受任务？non-goals 有没有被尊重？是否碰到 auth 或 data？测试跑了吗？有没有无关 cleanup？
别踩这个坑: 一个经常因为错误原因失败的 gate，很快就会变成背景噪音。gate 要少，但要可信。

文档与知识

以后还会被问到的决定，别让它死在聊天里。

很多好决定发生在 PR、会议、事故线程和聊天里。三个月后，大家只记得“当时好像讨论过”。文档要救的不是所有细节，而是未来的人真正需要的那一块。

先做这一步: 写下决策、取舍，以及什么时候要重新看。短一点，别写成论文。
趁热记录选文档类型写取舍链接源材料加重看条件

Docs-as-code

适合需要 review 和版本历史的工程文档。

团队知识库

适合跨职能背景和 onboarding。

决策记录

适合以后还会被追问的取舍。

AI summary 草稿

适合起第一稿，但事实要人来核。

决策：invite actions 使用 server-side permission checks。原因：UI checks 改善 UX，但不能作为授权边界。权限模型变化时重看。
别踩这个坑: 别把聊天总结直接发布成真相。summary 很容易把事实、猜测、owner 和开放问题揉在一起。

发布与变更管理

代码 merge 了，不等于变化已经安全。

这里从代码准备好之后开始。现在要判断谁会受影响，变化怎么到达他们，看什么指标，出问题怎么退。

先做这一步: 用受影响的人能听懂的话写 release note，不要照抄 commit。
确定读者列影响面选择 rollout定义健康指标写回滚条件

Release notes

适合用户可见变化、修复和 breaking behavior。

灰度发布

适合需要 flag、canary 或分阶段上线的高风险改动。

迁移计划

适合数据库、API、依赖升级或客户需要行动的变化。

发布观测

看 errors、latency、revenue、support volume 或 AI quality。

重复邀请处理先开给 10% workspaces。观察 invite API errors 和 support tickets。5xx rate 翻倍就关 flag。
别踩这个坑: Feature flag 不是免费的。每个 flag 都需要 owner 和清理时间。

场景教程

事故响应

事故里，清楚比聪明重要。

告警、聊天、dashboard、客户反馈和半成型猜测会一起涌过来。AI 可以帮你总结，但团队仍然需要一张共享图景，以及一个清楚的 owner。

先做这一步: 先建时间线：时间戳、事实、行动、owner、开放问题。
指定 owner建立时间线说明影响分开止血动作跟踪后续任务

Incident command

多人响应或客户受影响时用。

Observability triage

把 alerts、logs、traces 和最近 deploy 串起来。

Stakeholder 沟通

support、客户或管理层需要更新时用。

事故后学习

止血后，把预防动作变成可跟踪任务。

09:12 checkout 500 alert。09:18 找到最近 release。09:22 对 EU 关闭 checkout_coupon_v2。09:27 错误回到 baseline。后续补回归测试和 dashboard panel。
别踩这个坑: 不要让 AI 把猜测写成事实。时间线上的每一条，要么有证据链接，要么明确标成 hypothesis。

场景教程

团队 AI 治理

治理应该让好工作更容易，而不是让声音更大。

团队已经在编辑器、终端、浏览器、工单、文档和 PR 里用 AI。风险不是抽象的“用了 AI”，而是访问边界不清、ownership 不清、证据不清。

先做这一步: 盘点工具、数据访问、写权限，以及哪些动作必须人工批准。
盘点使用给数据分级拆开读写权限设置批准点事后回看规则

使用政策

定义数据分级、允许工具和 vendor 边界。

访问控制

管 repo、ticket、terminal、browser 和生产权限。

AI-assisted PR 标准

定义证据、ownership 和 review 预期。

学习闭环

根据真实事故和 near miss 更新规则。

AI 可以读批准过的 repo code、草拟 tests。push 代码、改生产设置、使用 customer data 前必须人工批准。每个 AI-assisted PR 都要有证据和 human owner。
别踩这个坑: 别从没人会用的大政策开始。先看真实 workflow，再围绕数据和写权限设边界。
