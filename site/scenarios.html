Scenario tutorials

Small workflows for real engineering situations.

Each tutorial gives you a problem shape, a first move, and a concrete example. Start with the closest scenario, then adapt it to your project.

Requirements to Tasks

Use this when a request is real, but still too vague for an engineer or AI assistant to start safely.

Walkthrough

  1. Write the user-facing change in one sentence.
  2. Name what is included and what is out of scope.
  3. Turn "done" into observable acceptance checks.
  4. Add clues about likely screens, APIs, data models, or commands.
  5. Decide how the work will be verified before coding starts.

Good first move

Create a short task brief with intent, non-goals, acceptance checks, constraints, and verification.

Intent:
Admins can invite one teammate by email.

Non-goals:
No bulk invite, role changes, or billing changes.

Acceptance:
Pending invite appears.
Duplicate invite shows an error.
Non-admins cannot invite.

Verification:
Run member tests and smoke test the browser flow.

Project Context Memory

Use this when AI keeps missing setup, conventions, or project shape.

Walkthrough

  1. List the last five things you had to explain repeatedly.
  2. Keep stable context and remove one-off task detail.
  3. Write a short guide near the repo.
  4. Link deeper docs instead of copying them.
  5. Update the guide when the same mistake repeats.

Good first move

Document run commands, test commands, important directories, conventions, and risky areas.

Local commands:
Install: pnpm install
Test: pnpm test
Typecheck: pnpm typecheck

Project map:
app/ product screens
api/ server routes
db/ schema and migrations

Careful:
Billing and membership need extra review.

Coding and Refactoring

Use this when AI can help write or move code, but the change still needs human control.

Walkthrough

  1. Start from a bounded task.
  2. Ask for a plan before edits.
  3. Work in reviewable checkpoints.
  4. Run tests, typechecks, lint, or smoke checks after meaningful steps.
  5. Review the final diff against the original task.

Good first move

Ask for a plan that names likely files, risk areas, and verification before code changes.

Task:
Move notification preference logic into a hook.

Plan:
Find current state and API calls.
Extract hook without changing behavior.
Update component.
Run notification tests.
Smoke test settings page.

Debugging

Use this when you have symptoms, but not a proven cause.

Walkthrough

  1. State the symptom in one sentence.
  2. Collect the strongest evidence.
  3. Write one hypothesis that predicts the next observation.
  4. Run a small experiment.
  5. Change code only after the cause is connected to evidence.

Good first move

Write a debugging note with symptom, evidence, hypothesis, and next check.

Symptom:
Checkout returns 500 after coupon is applied.

Evidence:
Server log shows currency_code is null.
Request includes coupon_id and currency=EUR.

Hypothesis:
Coupon recalculation drops currency_code.

Next check:
Trace applyCoupon to updatePaymentIntent.

Automated Verification

Use this when AI changed code and "it seems fine" is not enough.

Walkthrough

  1. Pick one behavior to protect.
  2. Choose the cheapest layer to test.
  3. Make the check fail on purpose once.
  4. Run it before AI changes related code.
  5. Save evidence such as trace, screenshot, log, or CI output.

Good first move

Automate or script one critical path before expanding coverage.

Behavior:
Admin can invite one teammate.

Check:
Open workspace settings.
Submit a valid email.
Confirm pending invite appears.
Confirm duplicate invite shows an error.

Evidence:
Trace or screen recording plus test output.

Code Review and Quality Gates

Use this when AI-generated work needs review before it becomes maintenance debt.

Walkthrough

  1. Compare the diff with the original task.
  2. Look for unexpected scope.
  3. Check tests and evidence.
  4. Review risky areas manually.
  5. Leave notes tied to behavior.

Good first move

Review task match and risk before style details.

Task match:
Does the diff implement the accepted task?
Are non-goals respected?

Risk:
Auth, permissions, billing, data, or migrations touched?
New dependency added?

Evidence:
Tests run?
Manual path checked?
CI linked?

Documentation and Knowledge

Use this when decisions and project knowledge keep disappearing into chat.

Walkthrough

  1. Capture the decision close to the work.
  2. Write the tradeoff behind the conclusion.
  3. Put it where people will look.
  4. Keep the note small.
  5. Add a maintenance hint.

Good first move

After meaningful work, write down the decision, tradeoff, and revisit condition.

Decision:
Use server-side permission checks for invites.

Why:
Client checks help UX, but cannot be trusted for auth.

Tradeoff:
UI hides actions, server remains source of truth.

Revisit when:
Permission model changes.

Release and Change Management

Use this when a change needs release notes, rollout steps, or rollback thinking.

Walkthrough

  1. Identify the audience.
  2. Summarize the change from merged work.
  3. Name the rollout path.
  4. Write the rollback path.
  5. Add watch signals.

Good first move

Write the user-facing change and rollback path before release.

User-facing change:
Admins can invite teammates from settings.

Rollout:
Internal workspaces first, all workspaces after 24 hours.

Watch:
Invite errors, email delivery failures, support tickets.

Rollback:
Disable workspace_invites flag.

Incident Response

Use this when scattered signals need to become a shared incident picture.

Walkthrough

  1. Build the timeline first.
  2. Separate facts from hypotheses.
  3. Track current owner and next action.
  4. Summarize customer impact.
  5. Draft the postmortem after the incident is stable.

Good first move

Collect timeline, known facts, open questions, and current owner.

Timeline:
10:42 error rate increases
10:45 alert fires
10:49 rollback starts
10:55 error rate normal

Known facts:
Checkout failed for EU users.
Recent deploy changed currency handling.

Open questions:
Why did tests miss this?

Team AI Governance

Use this when a team is already using AI tools and needs shared boundaries.

Walkthrough

  1. Inventory current AI use.
  2. Define data boundaries.
  3. Define write boundaries.
  4. Set review expectations.
  5. Revisit after real incidents or misses.

Good first move

Write what AI tools may read, what they may change, and what needs human approval.

AI may:
Read public docs and repository code.
Draft task briefs, tests, and PR summaries.

AI needs approval before:
Pushing code.
Changing production settings.
Using customer data.

Every AI-assisted PR includes:
Task summary, verification evidence, known limits.
