Human-first scenario tutorials

Find the stuck point first. Pick the tool later.

This is not a tool ranking or an AI planning spec. It is a working map for engineers: where you are stuck, what to do first, which route fits, and when to stop.

Requirements to tasks

The request is real, but still slippery.

Someone says, "Can we just add..." and everyone nods. Ten minutes later, the edge cases arrive: roles, empty states, billing, rollout, wording. This page helps you slow the work down just enough to make it implementable.

Do this first: Write one sentence that names the user-visible change. Then write what you are not doing.
Name the changeDraw the boundaryWrite checksPoint to likely filesPick verification

Tiny issue brief

Use it for small fixes and clear product changes. Keep it in the issue or PR.

PRD to task split

Use it when the direction is settled but engineering still needs slices.

Acceptance examples

Use it when roles, states, or edge cases can change the answer.

AI task shaping

Use it to find gaps, not to invent product decisions.

Admin can invite one teammate by email. No bulk invites, no role changes, no billing changes. Success means the pending invite appears and duplicate invite shows a clear error.
Watch the trap: The trap is writing a beautiful task that still hides the real decision. If pricing, permission, or rollout is undecided, say that before anyone writes code.

Project context memory

The assistant can read files, but it still misses the project.

If every new session starts with the same lecture about commands, folders, conventions, and dangerous areas, the problem is not the assistant. The project needs a small memory surface.

Do this first: List the last five things you had to explain twice. Keep the stable ones.
Collect repeated explanationsKeep stable factsPlace near the repoLink deeper docsUpdate after repeat mistakes

Repo guide

Use it for setup, tests, layout, conventions, and review expectations.

Codebase map

Use it when people get lost before they even know where to search.

Decision memory

Use it for architecture and product tradeoffs that will be questioned again.

Assistant context

Use it for team-approved rules, not personal prompts or secrets.

Install with pnpm install. Test with pnpm test. app/ holds product screens, api/ holds server routes, db/ holds schema and migrations. Billing and membership need extra review.
Watch the trap: Do not turn this into a dump of every private note. Shared guidance goes in the repo. Personal agent notes and credentials stay ignored.

Coding and refactoring

AI can move fast. Your job is to keep the change reviewable.

This is the page for implementation, cleanup, migrations, and refactors. The question is not "can AI write the code?" It usually can. The question is whether the resulting diff still has a shape a person can trust.

Do this first: Ask for a plan before edits. The plan should name files, risks, and checks.
Start from a taskApprove the planEdit in checkpointsRun cheap checksReview against the task

Pair assistant

Good for local edits, code search, small fixes, and test drafts.

Agentic loop

Good when a task needs multi-file edits plus command-line verification.

Codemod migration

Good when a repeated pattern is better changed mechanically.

Characterization first

Good when old behavior must not change.

Move notification preference logic into a hook. Do not change UI behavior, API shape, or persistence. Stop if the refactor needs API changes.
Watch the trap: The trap is letting a small refactor become a wandering cleanup tour. If the diff stops matching the task, stop and split it.

Debugging

Do not let a plausible story become a fix.

You have a symptom, maybe a stack trace, maybe a graph spike, maybe a user complaint. AI is useful here, but only if it helps you form testable hypotheses instead of decorating guesses.

Do this first: Write the symptom, the evidence, and one hypothesis that predicts the next observation.
State symptomCollect evidenceSeparate guessesTest a hypothesisAdd a regression check

Reproduce first

Use it when the failure can be made local or deterministic.

Observe first

Use it for production-only failures and distributed systems.

Change history

Use it when the failure follows a release, flag, config, or dependency change.

AI hypothesis pass

Use it to compare theories, not to accept the first one.

Checkout returns 500 after coupon apply. Logs show currency_code is null. Hypothesis: coupon recalculation drops currency for non-default currencies. Next check: trace EUR and USD fixtures.
Watch the trap: Changing three things at once feels productive. It also destroys the evidence trail.

Tutorial

Automated verification

Turn "seems fine" into evidence.

AI-assisted coding makes changes cheaper. That is good, until changes outpace proof. Verification is how you protect the few behaviors that would hurt if they quietly broke.

Do this first: Name one behavior you care about. Then choose the cheapest reliable check.
Name behaviorPick layerMake it fail onceRun in CI or locallyAttach evidence

Unit or component test

Best for rules, parsing, permissions, and UI states.

Browser or mobile E2E

Best for critical user flows that cross layers.

API or contract test

Best when the API is the product boundary.

AI eval

Best for RAG, agents, extraction, and variable outputs.

Admin invite flow: login as admin, open settings, submit email, confirm pending invite, repeat email, confirm duplicate error. Save the trace when it fails.
Watch the trap: E2E feels realistic, so teams overuse it. If a smaller test catches the same risk, start smaller.

Code review and quality gates

Review the intent before admiring the diff.

AI can produce a large, polished diff. That makes review harder, not easier. A good review starts with the task, then checks risk and evidence before style.

Do this first: Read the task brief first. Ask whether the diff still matches it.
Read taskCheck scopeScan riskCheck evidenceReview maintainability

Human review checklist

For behavior, scope, architecture, and judgment.

Static checks

For type, lint, security, dependency, and repeatable issues.

Build and test gates

For checks that should block merge.

AI-assisted review

For summaries and risk spotting, never final ownership.

Ask: did the diff implement the accepted task, respect non-goals, touch auth or data, run tests, and avoid unrelated cleanup?
Watch the trap: A gate that fails for bad reasons becomes background noise. Keep gates boring and trusted.

Documentation and knowledge

If a decision will be asked about again, rescue it from chat.

Teams make good decisions in PR threads, meetings, incidents, and chat. Three months later, nobody remembers why. Documentation is the act of saving the part future people will need.

Do this first: Write the decision, the tradeoff, and when to revisit it. Keep it short.
Capture close to workChoose doc typeWrite tradeoffLink source materialAdd revisit trigger

Docs-as-code

Good for engineering docs that need review and history.

Team knowledge base

Good for cross-functional context and onboarding.

Decision record

Good for tradeoffs people will question again.

AI summary draft

Good for first drafts, but facts still need a human.

Decision: invite actions use server-side permission checks. Why: UI checks help UX but cannot be the authorization boundary. Revisit when permissions change.
Watch the trap: Do not publish a chat summary as truth. Summaries blur facts unless someone checks names, dates, owners, and open questions.

Release and change management

A merged change is not the same as a safe change.

This page starts after the code is ready. Now you need to decide who is affected, how the change reaches them, what you will watch, and how you will back out if it goes sideways.

Do this first: Write the release note in the language of the affected person, not the commit.
Name audienceList affected surfacesChoose rolloutDefine health checksWrite rollback condition

Release notes

For user-visible change, fixes, and breaking behavior.

Progressive rollout

For risky changes that need a flag, canary, or staged release.

Migration plan

For database, API, dependency, or customer action.

Release observability

For errors, latency, revenue, support volume, or AI quality.

Enable duplicate invite handling for 10 percent of workspaces. Watch invite API errors and support tickets. Disable the flag if 5xx rate doubles.
Watch the trap: Feature flags are not free. Every flag needs an owner and a cleanup moment.

Incident response

During an incident, clarity beats cleverness.

Alerts, chats, dashboards, customer reports, and half-formed theories arrive at once. AI can help summarize, but the team still needs one shared picture and one clear owner.

Do this first: Create a timeline with timestamps, facts, actions, owners, and open questions.
Assign ownerBuild timelineState impactSeparate mitigationTrack follow-up

Incident command

Use it when multiple people are responding or customers are affected.

Observability triage

Use it to connect alerts, logs, traces, and recent deploys.

Stakeholder comms

Use it when support, customers, or leadership need updates.

Post-incident learning

Use it after mitigation to create tracked prevention work.

09:12 checkout 500 alert. 09:18 recent release found. 09:22 disable checkout_coupon_v2 for EU. 09:27 errors return to baseline. Follow up with regression test and dashboard panel.
Watch the trap: Do not let AI turn guesses into facts. Every timeline line should link to evidence or be marked as a hypothesis.

Tutorial

Team AI governance

Governance should make good work easier, not louder.

The team is already using AI: in editors, terminals, browsers, tickets, docs, and PRs. The risk is not AI in the abstract. The risk is unclear access, unclear ownership, and unclear evidence.

Do this first: Inventory tools, data access, write access, and where human approval is required.
Inventory useClassify dataSplit read/writeSet approvalsReview after incidents

Usage policy

For data classes, allowed tools, and vendor boundaries.

Access controls

For repo, ticket, terminal, browser, and production access.

AI-assisted PR standard

For evidence, ownership, and review expectations.

Learning loop

For updating rules after real incidents and near misses.

AI may read approved repo code and draft tests. It needs human approval before pushing code, changing production settings, or using customer data. Every AI-assisted PR needs evidence and a human owner.
Watch the trap: Do not start with a huge policy nobody uses. Start from actual workflows, then set boundaries around data and write access.
